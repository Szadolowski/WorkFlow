\subsection{Słownik głównych encji biznesowych}
Zaproponowany fizyczny model bazy danych charakteryzuje się wykorzystaniem unikalnych identyfikatorów \texttt{UUID} oraz rygorystyczną normalizacją struktur przestrzennych i kadrowych.

\begin{itemize}
    \item \textbf{User, Employee oraz BrigadeAssignment:} Architektura oddziela konto systemowe (\texttt{User}) od fizycznego profilu kadrowego (\texttt{Employee}). Tabela asocjacyjna \texttt{BrigadeAssignment} umożliwia elastyczne przypisywanie pracowników terenowych do konkretnych brygadzistów, zachowując pełną historię zatrudnienia.
    \item \textbf{Location, LocationAdministrator oraz Project:} Wydzielona struktura zarządzania placami budowy. Encja \texttt{Location} przechowuje dane adresowe. Tabela \texttt{LocationAdministrator} pozwala na precyzyjne przypisanie uprawnień (RBAC), definiując, który brygadzista ma dostęp do jakiego obszaru, a tabela \texttt{Project} powiązana jest z nią relacyjnie.
    \item \textbf{TimeEntry:} Kluczowa tabela ewidencji czasu pracy. Wykorzystuje typ wyliczeniowy \texttt{ENUM\_TimeEntryStatus} do kontroli maszyny stanów. Automatyzuje wyliczanie godzin na podstawie pól \texttt{startTime} i \texttt{endTime}.
    \item \textbf{Document:} Tabela integrująca aplikację z magazynem obiektowym MinIO. Przechowuje bezpieczny link (\texttt{fileUrl}), typ (\texttt{ENUM\_DocumentType}) oraz datę ważności.
    \item \textbf{SalaryRate oraz Payroll:} Implementacja historii zmian stawek za pomocą pól \texttt{validFrom} oraz \texttt{validTo}. Zamrożenie wypłaty i blokada edycji godzin odbywa się poprzez przypisanie \texttt{payrollId} (z tabeli \texttt{Payroll}) do rekordów czasu pracy w \texttt{TimeEntry}.
    \item \textbf{AuditLog:} Zaawansowany dziennik zdarzeń. Zapisuje akcje, powiązanych użytkowników oraz stan encji przed i po modyfikacji w formacie \texttt{JSONB} (\texttt{oldValue}, \texttt{newValue}), gwarantując audytowalność zgodną z RODO.
\end{itemize}

\subsection{Strategia komunikacji z warstwą danych}
Architektura aplikacji zakłada hybrydowe podejście do warstwy dostępu do danych (DAL -- \textit{Data Access Layer}). W celu zbalansowania tempa wytwarzania oprogramowania oraz maksymalnej wydajności dla skomplikowanych raportów biznesowych, przyjęto następujące ograniczenie architektoniczne:

\begin{itemize}
    \item \textbf{Zapytania ORM (50\% operacji):} Mapowanie obiektowo-relacyjne (np. \textit{Entity Framework, Hibernate} lub \textit{Prisma}) zostanie użyte do obsługi standardowych operacji CRUD (tworzenie użytkowników, proste edycje profili, dodawanie dokumentów). ORM zapewnia wysokie bezpieczeństwo przed atakami SQL Injection i przyspiesza cykl deweloperski.
    \item \textbf{RAW QUERY (50\% operacji):} Surowe zapytania w języku SQL zostaną wykorzystane w miejscach, gdzie wydajność i precyzja są krytyczne. Obejmuje to proces generowania skomplikowanych raportów płacowych (\texttt{Payroll}), które wymagają wielokrotnego łączenia tabel (JOIN) z zaawansowanym grupowaniem i filtrowaniem po dacie ważności stawek (\texttt{SalaryRate}). Wykorzystanie \textit{Raw Queries} gwarantuje optymalny czas wykonania zapytań do bazy zgodnie z wymogami niefunkcyjnymi ($<300$ ms).
\end{itemize}

\subsection{Implementacja fizyczna bazy danych (DDL)}
Poniżej przedstawiono fragment skryptu implementacyjnego, definiujący kluczowe tabele systemu wraz z więzami integralności oraz indeksami optymalizacyjnymi.

\begin{lstlisting}[language=SQL, caption=Definicja tabeli Pracownicy i Ewidencja Czasu]
-- Tabela ewidencji czasu pracy z mechanizmem automatycznego wyliczania
CREATE TABLE "TimeEntry" (
    "id" UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    "employeeId" UUID NOT NULL REFERENCES "Employee"("id"),
    "projectId" UUID NOT NULL REFERENCES "Project"("id"),
    "payrollId" UUID REFERENCES "Payroll"("id"),
    "date" DATE NOT NULL,
    "startTime" TIME NOT NULL,
    "endTime" TIME NOT NULL,
    -- Zaawansowana funkcja PostgreSQL - automatyczne wyliczanie przepracowanych godzin
    "hours" DECIMAL(5,2) GENERATED ALWAYS AS (
        EXTRACT(EPOCH FROM ("endTime" - "startTime")) / 3600
    ) STORED,
    "status" "ENUM_TimeEntryStatus" DEFAULT 'RAW',
    "createdBy" UUID NOT NULL REFERENCES "User"("id"),
    "createdAt" TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Indeks przyspieszajacy wyszukiwanie dokumentow po dacie
CREATE INDEX idx_doc_expiration ON Documents(expiration_date);
\end{lstlisting}

\subsection{Zaawansowane zapytania analityczne (Raw SQL)}
Zgodnie z przyjeta strategia, skomplikowane operacje finansowe realizowane sa poprzez surowe zapytania SQL. Ponizszy listing przedstawia proces generowania raportu płacowego, który uwzglednia historycznosc stawek (SCD Type 2).

\begin{lstlisting}[language=SQL, caption=Zapytanie wyliczajace wynagrodzenia za okres kwiecien 2026]
SELECT 
    e."firstName", 
    e."lastName", 
    SUM(t."hours") as total_hours,
    r."rate" as applied_rate,
    SUM(t."hours" * r."rate") as gross_salary
FROM "Employee" e
JOIN "TimeEntry" t ON e."id" = t."employeeId"
JOIN "SalaryRate" r ON e."id" = r."employeeId"
WHERE t."date" BETWEEN '2026-04-01' AND '2026-04-30'
  AND t."status" = 'APPROVED'
  -- Połączenie odpowiedniej stawki z datą wykonania pracy
  AND t."date" BETWEEN r."validFrom" AND COALESCE(r."validTo", '9999-12-31')
GROUP BY e."id", e."firstName", e."lastName", r."rate";
\end{lstlisting}
